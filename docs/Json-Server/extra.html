json-server 的進階運用

前言​

在介紹過基本應用後，接下來要來看一些進階的應用內容

自定義路由​

雖然 json-server 提供了很方便的 CRUD 路徑及分頁、篩選...等功能。但實際的 API 並不會像這樣在網址上操作
更可能得像下方表格一樣更簡單直覺地告訴前端，要怎麼去做 api 的 request

實際json-sever
/api/posts/1/posts/1
/posts/javascript/posts?category=javascript
/posts/javascript/2/posts?category=javascript&_page=2

所以 json-server 也提供了一個自定義路由的功能來達成上述的問題。

{ "key": "value" } // key 的部分為 request 的網址， value 則為轉換成 json-server 認識的網址

舉例而言

{ "/api/*": "/$1" }

$1 的做法就是讓省略掉上層的路徑，例如 /api/posts/ 傳進來經過處理就會變成 /posts/ ， 能夠符合 json-server 的路由規則

{ "/posts/:category": "/posts?category=:category" }

這邊有使用到動態參數的功能 (冒號:category)，他可以讓接收到的參數同時用於 key 及 value
例如 /posts/javascript 經過處理就變成 posts?category=javascript ，藍色的部分就是動態參數

{ "/:resource/:category": "/:resource?category=:category" }

而動態參數也可以運用在大部分規則都相同下的路由，就可以不用 posts 寫了一次篩選規則，todos 再寫一次， comments 又寫了一次

/posts/javascript ：/posts?category=javascript
/todos/works ：/todos?category=works

最終我們把所有的條件都彙整到 route.json 檔案下

route.json
{
  "/api/*": "/$1",
  "/posts/:category": "/posts?category=:category"
}

完成設定後，啟動時需要額外設定路由檔

json-server --watch db.json -p 3001 -r route.json // -r 帶入路由設定檔

自定義處理​

雖然在送出及獲得資料上沒問題，但 json-server 回傳的內容相對也簡易許多，回傳送出資料或是失敗則為空物件 {}
實際應用中的 API 卻不是這樣，往往可能都會帶有 Respones Code 來讓前端判斷是否正確，若失敗則要顯示失敗訊息

而在傳送資料的時候也可能會遇到錯誤。 舉例來說：

後端的欄位尚未填寫完整
成立訂單時未寫入收件人資料導致訂單成立後不知道寄給誰，這時候後端在收到資料時就要先擋下提醒，而不是繼續成立訂單。

這時候在 json-server 上模擬就可以使用到 middleware 的功能

middleware 用來自定義 json-server 在收到 request 後，修改 db.json 前中間做的事情，例如欄位、Token 檢查是否正確... 等等。

直接建立一支 Posts.js

Posts.js
// req : request 過來的資料 (header token & body 的 JSON 資料都在這裡)
// res : response 回去的資料
// next : json-server 的 function，在完成自定義 function 後執行讓後續流程繼續執行
module.exports = (req, res, next) => {
  // 要做的事情 ...
};

檢查欄位是否填寫​

Posts.js
module.exports = (req, res, next) => {
  let data = req.body; // 取得送來的 JSON 資料
  if (req.url.indexOf("/posts/") >= 0 && !data.title) {
    // 若 request 為 posts 及 body 未有 title 資料
    res.json({ success: false, message: "未輸入標題內容" }); //回傳錯誤訊息
    return; // 這邊要記得終止 function，不然還是會做 db.json 資料修改
  }
  next(); // 若有 title 資料則繼續往下走，新增 post，若沒有執行 next 會沒有回應而 timeout
};

完成 middleware 的內容後要我們重啟 json-server，並告訴它我們要使用自己的 middleware

json-server --watch db.json -p 3001 -m Posts.js // -m 可帶入多個檔案 ex: -m Posts.js Users.js

這時候回 postman 重新 POST 一筆新的 posts，但不傳送 title，送出後則回傳錯誤訊息，這樣就成功做到檢查 request 這段了

檢查 Token 是否正確​

而除了欄位檢查外，有些需要登入權限才能做的事，例如新增文章、加入購物車、取得購物車項目等，都會去做額外的檢查
檢查的內容可以用 cookie 或是 JWT 的 Token，這邊舉例而言要檢查是否有帶入 Token

Posts.js
module.exports = (req, res, next) => {
  const token = req.header.authorization;

  if ((req.url.indexOf("/posts/") >= 0 && !token) || !verifyToken(token)) {
    // 若 request 為 posts 及未夾帶 authorization 資料或驗證未通過
    res.status(401).json({ success: false, message: "Token 錯誤" }); // status 可以設定回傳的 status code
  }

  next(); // 若有 Token 及驗證成功，新增 post
};

這時候可以看到除了 message 顯示的內容不一樣外，上方的 200 也變成 401 Unauthorized

info

延伸閱讀： HTTP response status codes，於不同的情境回傳不同的 status code

自定義回傳內容​

剛剛已經介紹過自定義處理使用 middleware，接下來則是 response 的部份
從 初入 json-server 中我們已經知道，json-server 會回傳對應的 object 給我們
剛剛在錯誤處理這段回傳內容是帶有 success 和 message 的 object，若成功時沒有 success 的話前端會比較難判斷到底有沒有成功

不過以上使用的都是 json-server-cli 中的指令快速去建立環境，如果要做到自定義回傳的話就要用另一個方式來建造這個環境。

以非 json-server-cli 方式建立 json-server​

首先先新增一支 Server.js

Server.js
const jsonServer = require("json-server");
const server = jsonServer.create();
const router = jsonServer.router("db.json");
const PORT = 3001;

server.use(jsonServer.defaults()); // 載入 json-server 的基礎設定

// 將 middleware 直接寫進來
server.use(jsonServer.bodyParser); // 要 use(jsonServer.bodyParser) 才能取得 request 的資料

// 還可以使用 server.get("...") server.put("...") server.patch("...") server.delete("...")
server.post("/posts/", (req, res, next) => {
  // 與剛剛額外拆出去的 Posts 一樣，只不過就不用特地判斷 req.url
  let data = req.body;
  if (req.url.indexOf("/posts/") >= 0 && !data.title) {
    res.json({ success: false, message: "未輸入標題內容" });
    return;
  }
  next(); // 一樣要記得 next 不然會卡在這裡;
});

server.use(
  jsonServer.rewriter({
    // 這邊設定自定義路由，要在 server.use(router) 前
    "/api/*": "/$1",
    "/posts/:category": "/posts?category=:category",
  })
);

server.use(router); // use(router) 要放在 middleware 後面才會執行 middleware

// 這段是最終 response 的覆寫，除了原本的資料外還會新增 success: true;
router.render = (req, res) => {
  res.json({
    success: true,
    message: res.locals.data,
  });
};

server.listen(PORT, () => {
  // 等同於剛剛 json-server  db.json -p 3001
  console.log(`JSON Server is running at http://localhost:${PORT} ~`);
});

重新執行 json-server

// 以 node 方式執行而非 json-server ...
node Server
// JSON Server is running at http://localhost:3001 ~

這時候再來重新發送 request，登愣~ 就能看到這次被覆寫的 response

caution

不過前提就是在自定義處理 middleware 這塊要處理好，不然都會出現 suceess : true
