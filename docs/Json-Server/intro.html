初入 json-server

前言​

在前後端分離的專案下，往往會碰到前端進行開發時，後端尚未提供相關的 API 使用的時間差，
這時候就能透過 JSON Server 來做模擬，亦也適合初入串接 API 的前端來熟悉 API 的使用方法。

需要用到​

  • Node.js
  • Postman

安裝 json-server​

npm install -g json-server // 安裝 json-server

待安裝結束後，需要在該目錄下新增一個 db.json 檔案供 json-server 產生 API 路徑

note

若不安裝的話就使用 npx 來完成指令

{
  "posts": [{ "id": 1, "title": "json-server", "author": "typicode" }],
  "comments": [{ "id": 1, "body": "some comment", "postId": 1 }],
  "profile": { "name": "typicode" }
}

按官方提供範例來看，json-server 會依照上方資料自動產生三個路徑分別是根部的三個資料 post comments profile

tip

db.json 檔案皆可自行命名，隨後記得更改為自己命名的名稱即可。

這時候透過 cmd 執行指令啟動 json-server

json-server --watch db.json
tip

json-server 預設的 port 為 3000，但常常會與其他開發環境衝突，若需要更改於指令後方加入 --port [port] 即可

json-server --watch db.json -p 3001

終端機出現 json-server 的歡迎訊息就表示成功了

  \{^_^}/ hi!

  Loading db.json
  Done

  Resources
  http://localhost:3001/posts
  http://localhost:3001/comments
  http://localhost:3001/profile

  Home
  http://localhost:3001

  Type s + enter at any time to create a snapshot of the database
  Watching...

json-server 首頁​


在 Resources 下方可以看到依照 db.json 產出的內容

  • /posts /comments 下有一筆資料
  • /profile 則是會回傳一個 object 物件

基本使用方式​

現在有了路徑之後，就可以透過 postman 軟體來進行 API 的 request 測試


再執行 postman 後，首頁右方有一個 + 的按鈕，按下後就會產生一個空白的 request
產生後如同上方畫面，在 URL 的左方有一個下拉選單提供所有 request 的 methods
選擇 method 同時填入 URL 後即可送出 request

GET - 取得資料​

而回到 json-server 來，使用 GET 來執行 request 時則會回傳 db.json 的值

進階使用

  • id 特定資料
  • Filter 篩選資料
  • Paginate 分頁
  • Sort 排序
  • Slice 取得特定範圍
  • Operators 比較運算子
在剛剛預設的資料中，有發現每筆資料都會有一個 id，用途就是提供使用者輸入正確的參數後能夠得到正確想要的資料
所以每一個 id 都需要是唯一值，即使我的資料有 100 筆，但我正確輸入我要用的 id 就只會拿掉這筆資料
常見會用在文章、產品中，用來取得要用的這篇。
-GET http://localhost:3001/posts/1 // 路徑後方斜線 id
除了需要特定資料外，也可能需要大量的特定資料，這時候 id 就顯得不方便許多
所以就可以使用篩選的功能來達成。常會用在部落格要篩選出某種分類下文章，或是論壇上某些作者的文章。
-GET http://localhost:3001/posts?id=1&id=2 // 除了網址常不方便，還要取得所有的 id 
-GET http://localhost:3001/posts?author=typicode // 路徑後方問號待參數 欄位=值
有時候資料量真的太大，不太可能一次就透過 API 的方式將幾萬筆的資料取回來
這時候分頁功能就是最好的朋友了，透過參數功能 _page(第幾頁) 及 _limit (每頁筆數)
這個功能常見會運用在搜尋上，若是使用者輸入的關鍵字真的範圍太廣，就可以靠分頁慢慢吐回搜尋的資料
-GET http://localhost:3001/posts?_page=2 // 如果沒有帶入 _limit 預設為 10筆 
-GET http://localhost:3001/posts?_page=2&_limit=20
使用的情境一樣是在有大量資料的情況下，可以透過特定的排序條件，讓想被看到的資料可以排在前面
會有兩個參數可以使用 _sort(用這個欄位來排序) 及 _order (排序的方式 asc 小到大 desc 大到小)
-GET http://localhost:3001/posts?_sort=views&_order=desc // 文章中點擊最高排到最低
在其他相較之下較有條理的分類篩選下，也有這種可以取得特定範圍資料的做法
其實也就等同於 js 中的 Array.slice
會用到的參數有三個 _start (起始) _end (結束) _limit (筆數)
可以用在如果要在每次首頁都顯示不同的文章或產品時，就可以亂數產生_start和_end
-GET http://localhost:3001/posts?_start=20&_end=30
前面的功能都是特定條件，那如果遇到較大範圍條件的話就要透過 Operators 達成了
  • _gte大於等於 _lte小於等於
  • _ne不等於
  • _like模糊搜尋或可用正規表達式
-GET http://localhost:3001/posts?id_gte=3 // id 大於等於3的資料
-GET http://localhost:3001/posts?id_ne=3 // id 不等於3的資料
-GET http://localhost:3001/posts?title_like=json // 標題有關 json 的資料

POST - 新增資料​

使用 POST 可以針對更路徑下新增一筆資料，除了 method 選擇 POST 外
所要新增的資料要放於 Body 之中的 raw 並選擇 JSON 格式

// -POST http://localhost:3001/posts/
{
  "title": "First Posts",
  "author": "WeiJie"
}
// Respones
{
  "title": "First Posts",
  "author": "WeiJie",
  "id": 2 //id 的部分會由 `json-server` 自行新增
}

PUT 、 PATCH - 修改資料​

使用 PUT PATCH 都是可以針對特定 id 資料進行修改
差別則為 PUT 會做 整筆資料的取代， 而 `PATCH` 只會修改 特定的欄位 

// -PUT http://localhost:3001/posts/2
{
  "title": "First Posts Modify"
}
// Respones
{
  "title": "First Posts Modify",
  "id": 2 
}

// -PATCH http://localhost:3001/posts/2
{
  "title": "First Posts Modify"
}
// Respones
{
  "title": "First Posts Modify",
  "author": "WeiJie",
  "id": 2
}

DELETE - 刪除資料​

在查詢新增修改後，就只剩下刪除了，一樣需要帶入 id 欄位，完成後則傳回空字串 

// -DELETE http://localhost:3001/posts/2

// Respones
{}

Request 成功但資料沒有更新​

danger

若 POST PUT PATCH 於實際使用時不成功，請檢查 request 的 header 中是否設定 Content-Type: application/json
若未設定 request 還是可以順利發送，但實際上的資料卻不會更新
postman 預設都是帶上的，但開發上使用 fetch 尚需留意

測試資料集使用 - JSONPlaceholder​

如果測試上自己一筆一筆新增資料是真的太慢了，這時候就可以使用網路上提供的測試資料 JSONPlaceholder

提供了以下內容可以使用

  • /posts
  • /comments
  • /albums
  • /photos
  • /todos
  • /users

可以直接使用 postman request 做上述的操作

//-GET https://jsonplaceholder.typicode.com/posts

// Respones
{
  [
    {
        "userId": 1,
        "id": 1,
        "title": "sunt aut facere repellat provident occaecati excepturi optio reprehenderit",
        "body": "quia et suscipit..."
    }, 
    ....
}
