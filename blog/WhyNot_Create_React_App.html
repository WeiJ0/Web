為什麼不再使用 create-react-app

November 24, 2022 · 4 min read
WeiJie

前言​

以往最開始學 React 的時候，起手式都是 create-react-app (文中會簡稱為 CRA)

它的出現簡化了再專案最初期建置的時間，透過 cmd 就可以快速建起一個可以運行的 React 專案

再初期學習的我們能夠快速看到一個結果，一定會信心大增。但隨著程式越寫越多，專案越來越大時
會發現每次執行 dev 指令等待的時間會越來越久。

原因是 CRA 透過 webpack 來 bundle 你的專案，而 webpack 會將你的專案打包成一個檔案，這個檔案會包含你的所有程式碼，包含你的第三方套件，這樣的好處是可以讓你的專案在瀏覽器上執行，但是缺點是每次你修改程式碼，都會重新打包，這樣就會花費更多的時間等待。

Vite​

Vite 基於 esbuild，它的出現就是為了解決 webpack 打包的缺點，它不會將你的專案打包成一個檔案，而是將你的專案分成多個檔案，這樣就不會需要重新載入所有的程式碼
底層的部份由 Go 編寫，網路上其他大神實測出來，速度比 webpack 快 10 倍以上

而除了 React 之外，Vite 也支援 Vue、Svelte 等等的框架，所以成為越來越多工程師首選的開發工具
從 npm 的每週下載量及趨勢圖來看，兩者有著不小的差異

Turbopack​

而到了 2022 年的 10 月， Next.js 框架背後的 Vercel 公司公布一款號稱比 Vite 更快的工具降世了，它就是 Turbopack， 這次 Turbopack 則是透過 Rust 開發，雖說官方數據能比 Vite 再快上 10 倍，實際上也引發了網路上眾多開發者的質疑。

在實質數據上，在含有 3000 模組的專案中，Turbopack 花了 1.8 秒就可以啟動，而 Vite 花了 11.4 秒。

其背後與 vite 的差異除了語言外再者就是運用了 cache 做處理，esbuild 沒有做太多的 cache 處理，所以會執行同樣的工作非常多次，而導致整體速度下降。 不過雖說如此但除非這開發到像是 3000 模組這種大專案，不然再感受上來說應該不會有太大的差異。

心得​

而在看完這場前端工具大戰及網路上眾多測試和評論後，我認為還是回歸最原始
這些工具開發的目的都是讓開發者能夠更專注於程式碼的開發上
當然各個工具都有各自的擁護者，開發上選擇自己最熟悉使用的工具維護專案上才能有最好的效率
